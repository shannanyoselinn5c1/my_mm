1. Общие положения и определения
1.1. Внешний образ
Докер образ, получаемый из внешнего реестра образов - hub.docker.com, mcr.microsoft.com и т.п.
1.2. Кастомный образ
Докер образ, описываемый Dockerfile'ом в репозтории.
1.3. Безопасность
Мы стремимся обеспечить безопасность наших серверов и сторонников. Для того, чтобы снизить поверхность атаки и потенциальный ущерб мы должны запретить нашим приложениям делать то, что им не требуется делать. Например, запретить сервису БД обращаться к интернету.
1.4 Трудоемкость
Мы стремимся снизить трудозатраты деплоя. Это требует унификации процесса и автоматизации. Ручные операции при деплое должны быть сведены к минимуму.
1.5. Исключения
Каждое отклонение от правил требует отдельного согласования.
2. Основной инструмент и структура репозитория
2.1. Инструмент
К деплою принимаются только приложения, оформленные для разворачивания с помощью docker compose, даже если приложение состоит только из одного сервиса.
2.2. Директория deploy 
В корне репозитория должна располагаться директория deploy.
2.3. Содержимое deploy 
Директория deploy должна содержать следующие файлы: Dockerfile, docker-compose.yml, .env и, возможно, README.md. Других файлов в директории deploy быть не должно.
3. Dockerfile
3.1. Расположение
Файл должен располагаться в репозитории по отноительному пути deploy/Dockerfile
3.2. Содержание
Dockerfile должен содержать инструкции для всех кастомных образов, используемых приложением.
3.3. Контект
В качестве контекста для сборки Dockerfile должен расчитывать на корень репозитория. Следовательно, все пути для директивы COPY должны быть указаны относительно корня репозитория. Например, собрать образ background-fetcher из примера выше должно быть можно следующей командой, выполненной из корня репозитория:
docker build --tag hub.xtra.vg/my-app/background-fetcher --target background-fetcher --file deploy/Dockerfile .
3.4. FROM.image
image в директиве FROM должен иметь фиксированную версию.
4. docker-compose.yml
4.1. Расположение
Файл должен располагаться в репозитории по отноительному пути deploy/docker-compose.yml.
4.2. Внешние образы
Внешний образ должен иметь фиксированную версию.
Сервис с кастомным образом должен иметь image установленного формата.
Формат: hub.xtra.vg/{application-name}/{image-name}, где
    • {application-name} - англояычное название приложения, совпадает со slug репозитория. Например, для репозитория https://git.xtra.vg/rtf/it-workshop/bots/mattermost/editor-duty-bot равен editor-duty-bot. 
    • {image-name} совпадает с названием target в Dockerfile 
Директива build сервиса не должна использоваться.
4.4. Restart
restart сервиса должен иметь значение always.
4.5. Dependencies
depends_on сервиса должен быть корректно заполнен. Например, если приложение требует для своей работы БД, то у приложения должна быть прописана зависимость от БД.
4.6. User
user сервиса должен иметь значение nobody.
4.7. Сеть
Только сервисы, которым требуется взаимодействовать по сети должны быть присединены к одному network.
Допустим у нас есть три сервиса - два прикладных приложения appA и appB и база данных db. Архитектура приложения не предполагает взаимодействия appA и appB. Поэтому при деплое мы должны сделать так, чтобы appA и appB не были связаны одной сетью. По умолчанию docker compose создает сеть, к которой подключает все сервисы приложения. 
4.8. Сеть. Интернет
Доступ в интернет должен быть только у сервисов, которым для работы требуется доступ в интернет или которые выставляют (expose, ports) порты на хостовый интерфейс. Отключить доступ в интернет можно, подключив сервис только к internal сетям. 
Альтернативный вариант отключения доступа в интернет состоит в задании директивы network_mode сервиса в значение none.
4.7. Неизменяемость контейнера
read_only сервиса должен иметь значение true.
4.8. Volumes
Вольюмы сервиса должны быть bind mount'ами директорий. Все bind mounts на хосте должны быть расположены в директории ./volumes.
4.9. Volumes.Mode
Режим доступа сервиса к вольюму должен быть указан.
rw вольюмы допустимы только тогда, когда сервису требуется записывать в вольюм
4.11. Переменные окружения
Сервис должен видеть только те переменные окружения, которые ему требуются для работы.
Все переменные окружения приложения должны быть перечислены в файле .env.example (см. раздел 5), при этом директиву env у сервиса указывать не надо - docker compose использует файл .env по умолчанию. Разграничение видимости переменных сервисами следует задавать через директиву environment:
4.12. Порты
Публиковать порты через директиву ports сервиса на хост можно только на 127.0.0.1.
4.13. Порты. Допустимость
Публиковать порты сервиса на хост можно только если это необходимо для работы приложения.
4.14. CPUS
Директива cpus сервиса должна быть задана. Разрешено не более 1 для сервиса и не более 3 суммарно на приложение.
4.15. RAM
Директивы mem_limit и memswap_limit сервиса должны иметь одинаковое значение. Разрешается не более 1 ГБ на сервис и не более 3 ГБ суммарно на приложение.
Равенство mem_limit и memswap_limit не позволит приложению уходить в своп и таким образом приводить всю систему к деградации. https://docs.docker.com/config/containers/resource_constraints/#prevent-a-container-from-using-swap.
4.16. Другие разрешенные директивы
Разрешено использовать директивы сервиса cap_drop, command, healthcheck, pids_limit, expose, post_start, pre_stop, stop_grace_period, stop_signal, dns_search, working_dir.
4.17. SECURITY_OPT
Каждый сервис должен содержать директиву security_opt следующего содержания:
security_opt:
  - no-new-privileges=true
4.18. tmpfs
Для tmpfs сервисов нужно обязательно указывать:
    • size не больше 512 мегабайт 
    • noexec 
    • nosuid 
4.19. Прочие директивы
Директивы сервисов, не перечисленные здесь, запрещены к использованию.
5. .env.example
5.1. Расположение
Файл должен располагаться в репозитории по отноительному пути deploy/.env.example.
5.2. Содержимое
5.2. Все переменные окружения, требуемые сервисам приложения, должны быть перечислены в файле .env.example.
5.3. Формат
Файл должен иметь формат env файла.
5.4. Значения
В качестве значения для переменной должно выступать либо ее умолчательное значение, либо описание назначения переменной на русском языке.
Например:
POSTGRES_HOST=postgresdb
POSTGRES_DB=rtfdb
POSTGRES_USER=postgres
POSTGRES_PASSWORD=postgres
MM_URL=https://talk.rotfront.org
MM_TOKEN=токен для авторизации бота
MM_TEAM=rotf MM_CHANNEL=название канала в маттермосте
6. Ручные операции и бутстраппинг
6.1. Схема БД
Схему БД должно создавать само приложение. Если схема меняется от релиза к релизу, то приложение должно само мигрировать без участия администратора.
6.2. Ручные операции
Если ручного бутстрапа не избежать, то можно создать специальный сервис для бутстрапа с четкими инструкциями для администратора. Инструкции должны быть зафиксированы в файле deploy/README.md.
6.3. Логи
Сервисы должны писать логи на консоль.
6.4. Сожеримое логов
Записи в логах должны содержать временную метку.
7. Базовые образы и сборка образов приложений
7.1. Большие файлы
Большие файлы желательно скачивать на этапе построения образа. К ним относятся словари, LLM модели, бинари и т.п.
7.2. Сигналы
Контейнеры сервисов должны оперативно реагировать на сигналы - SIGTERM и т.п. Здесь полезно изучить https://docs.docker.com/reference/dockerfile/#shell-and-exec-form.
